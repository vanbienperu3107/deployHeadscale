Ping-reporter — cách chạy, sequence & call flow

Cơ chế đo độ trễ giữa các node trong tailnet, và vì sao dashboard /derp đôi khi chỉ hiện vpn2 còn vpn3/vpn4/vpn5/vpn6 báo ping-reporter chưa chạy.  ·  Bản đọc offline; nguồn chuẩn là PING-REPORTER.md.

0 · TL;DR — vì sao chỉ có vpn2 có số?

Bảng “Ping từ từng server → node” gộp 2 nguồn khác nhau vào chung bảng SQLite node_latency:

Nguồn hiển thịAi tạo raChạy ở đâuPhụ thuộc gì
vpn2node-dedup tự ping (SRC_NAME=collector)trên vpn2luôn có, không cần reporter
vpn3 / vpn4 / vpn6container ping-reportertrên chính node đócần reporter chạy + POST về được collector
vpn5(chưa có)—repo relay-vpn5/derp-vpn5 chưa có service ping-reporter
vpn2 luôn có số vì đó là collector tự đi ping mọi node qua LocalAPI của sidecar. Các hàng còn lại trống khi reporter trên node đó chưa chạy / chưa POST được về collector. vpn5 sẽ luôn trống cho tới khi thêm service reporter.

Một hàng (vd vpn3) chỉ có số khi đủ 5 mắt xích:

  • container ping-reporter-vpn3 đang chạy trên vpn3
  • sidecar ts-vpn3 đã join tailnet (có IP 100.64.x.y)
  • reporter tìm thấy peer tên collector ONLINE qua LocalAPI
  • ts-forward (socat trong netns collector của vpn2) còn sống
  • node-dedup trên vpn2 còn sống

1 · Kiến trúc tổng quan

Hai chiều đo đổ vào cùng một bảng node_latency, rồi dashboard đọc ra.

Relay node (vpn3 / vpn4 / vpn6)

ping-reporter reporter.py · chiều 1
│ LocalAPI qua unix socket ▼
sidecar tailscaled ts-vpn3 · có IP tailnet

vpn2 — control + collector

ts-forward (socat) listen :8090 trong netns collector (100.64.0.1)
▼
node-dedup dedup.py · HTTP :8090 + self-ping (chiều 2)
▼ ▲ self-ping qua LocalAPI sidecar 'collector'
SQLite node_latency / devices
ping-reporter ══ POST /metrics/report → 100.64.0.1:8090 ══▶ ts-forward → node-dedup
Vì sao cần ts-forward? node-dedup đã tách khỏi netns sidecar (chạy container riêng để /stats luôn sống kể cả khi sidecar restart) nên không còn IP tailnet. ts-forward (socat) chạy trong netns sidecar collector (IP tailnet 100.64.0.1), lắng nghe :8090 rồi forward sang node-dedup:8090. Reporter POST tới 100.64.0.1:8090 → tới được node-dedup.

2 · Sequence — một vòng poll của reporter (mặc định 30s)

reporter→LocalAPI
GET /localapi/v0/status — lấy danh sách Peer (Online, IP, HostName)
reporter
get_peers_and_collector() — tách peers ONLINE + tìm IP node collector (ưu tiên bản ONLINE, bỏ bản trùng OFFLINE)
Nếu không thấy collector ONLINE → log “collector chua thay…” → ngủ, thử lại vòng sau.
reporter→LocalAPI → peer
POST /localapi/v0/ping?ip=<peer>&type=disco — lặp cho từng peer online
LocalAPI→reporter
LatencySeconds, path = direct nếu có Endpoint, ngược lại derp:<code>
reporter→ts-forward
POST /metrics/report {hostname:"vpn3", samples:[…]} → 100.64.0.1:8090
node-dedup→SQLite
_authed() → validate_report() → record_report() → INSERT node_latency (src='vpn3')
node-dedup→reporter
200 {ok:true, stored:N} → reporter log “[vpn3] ping k/n OK → collector …: OK”

Dashboard đọc độc lập (tự refresh 30s): trình duyệt → Caddy(+SSO) → node-dedup → query_all_server_pings()/query_latency() → SQLite → HTML bảng.

4 · Cấu hình mỗi node

Mỗi relay node khai báo service ping-reporter trong compose của nó (derp-vpn3/, derp-vpn4/, relay-vpn6/):

ping-reporter:
  image: python:3.12-slim
  container_name: ping-reporter-vpn3        # đổi theo node
  restart: unless-stopped
  network_mode: "service:tailscale"          # BẮT BUỘC: chung netns sidecar
  command: ["python3", "/app/reporter.py"]
  environment:
    - REPORTER_NAME=vpn3                      # PHẢI khớp DERP region code
    - TS_SOCKET=/var/run/tailscale/tailscaled.sock
    - POLL_INTERVAL=30
    - COLLECTOR_PORT=8090
    - PING_TIMEOUT=8
  volumes:
    - ../ping-reporter/reporter.py:/app/reporter.py:ro
    - ts_sock:/var/run/tailscale                # socket LocalAPI chia sẻ
  depends_on: [ tailscale ]
  • network_mode: "service:tailscale" — reporter dùng chung netns sidecar nên có IP tailnet để POST qua WireGuard. Thiếu → POST timeout.
  • REPORTER_NAME phải trùng DERP region code (vpn3/vpn4/vpn6) thì dashboard mới ghép đúng hàng.
  • ts_sock — socket tailscaled.sock chia sẻ giữa sidecar và reporter.

6 · Di chuyển sang server mới (server migration)

6.1 Chuyển một relay node (vd vpn3) sang VPS mới

  1. DNS vpn3.hangocthanh.io.vn trỏ về IP VPS mới (relay/derper tự xin lại cert).
  2. Chạy workflow deploy của node (hoặc docker compose up -d --build trong derp-vpn3/). Service ping-reporter tự khởi động lại.
  3. Không cần đổi gì ở collector: reporter tìm collector theo hostname collector, không hardcode IP. REPORTER_NAME=vpn3 giữ nguyên → ghép đúng hàng.
  4. Xác minh: docker logs ping-reporter-vpn3 thấy “→ collector …: OK” và /metrics/health?expect=vpn3 trả ok:true.

6.2 Chuyển collector / vpn2 sang server mới

  1. Lịch sử latency & MAC nằm trong volume dedup_data (/data/devices.db) — copy volume nếu muốn giữ lịch sử; không copy thì DB khởi tạo trống (reporter điền lại sau vài vòng).
  2. vpn2 mới chạy: sidecar ts-collector join lại hostname collector, ts-forward mở :8090, node-dedup lắng nghe.
  3. Reporter các node không cần sửa: tự khám phá lại peer collector ONLINE và POST về IP tailnet mới (đã ưu tiên bản ONLINE nếu còn bản cũ OFFLINE).
  4. node-dedup tự gom/xoá node collector trùng OFFLINE.
